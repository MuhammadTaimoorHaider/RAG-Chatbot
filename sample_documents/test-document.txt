RAG Chatbot System - Technical Documentation

INTRODUCTION

This document provides a comprehensive overview of the RAG (Retrieval Augmented Generation) chatbot system. The system was developed to demonstrate production-ready machine learning engineering practices while maintaining zero operational costs.

SYSTEM OVERVIEW

The RAG chatbot is an intelligent question-answering system that retrieves relevant information from uploaded documents and generates accurate, source-cited responses. Unlike traditional chatbots that rely solely on pre-trained knowledge, this system grounds its responses in user-provided documents, significantly reducing hallucinations and improving accuracy.

ARCHITECTURE

The system employs a microservices architecture with three distinct layers:

Frontend Layer:
The user interface is built with Streamlit, providing an intuitive chat-based interface. Users can upload documents, ask questions, and view conversation history. The interface displays source citations for each response, allowing users to verify information.

Backend Layer:
FastAPI serves as the REST API backend, exposing endpoints for document upload, chat queries, health checks, and conversation management. The backend orchestrates the RAG pipeline, coordinating between document processing, vector search, and language model inference.

Data Layer:
Pinecone stores document embeddings in a vector database optimized for similarity search. Redis maintains conversation history with automatic TTL management. The system gracefully falls back to in-memory storage if Redis is unavailable.

TECHNICAL IMPLEMENTATION

Document Processing:
Documents are processed through a multi-stage pipeline. First, format-specific loaders extract text content. The RecursiveCharacterTextSplitter then chunks documents into 1000-character segments with 200-character overlap to maintain context across boundaries. Each chunk includes metadata such as source filename, page number, and chunk identifier.

Embedding Generation:
The system uses the HuggingFace all-MiniLM-L6-v2 model to generate 384-dimensional embeddings. This model runs locally, eliminating API costs while maintaining good performance. The embeddings capture semantic meaning, enabling accurate similarity search.

Vector Search:
When a user asks a question, the query is embedded using the same model. Pinecone performs cosine similarity search to retrieve the top-K most relevant chunks. The default configuration retrieves 4 chunks, balancing context richness with response coherence.

Language Model Integration:
Groq provides access to Llama 3.3 70B through their free API. Groq's optimized inference infrastructure delivers sub-second response times. The model receives the retrieved context, conversation history, and user query to generate informed responses.

Conversation Memory:
Redis stores conversation history organized by session ID. Each session maintains up to 10 messages with automatic expiration after 1 hour. This enables context-aware responses while managing memory efficiently.

KEY FEATURES

Multi-Format Support:
The system handles PDF, DOCX, TXT, and Markdown files. Format-specific loaders ensure proper text extraction regardless of document structure.

Conversational Context:
Unlike single-turn question answering, the system maintains conversation state. Users can ask follow-up questions that reference previous context, creating natural dialogue flows.

Source Attribution:
Every response includes citations showing which document chunks informed the answer. This transparency builds user trust and enables fact verification.

Production Readiness:
The system includes health checks, structured logging, error handling, and graceful degradation. Docker containerization ensures consistent deployment across environments.

API-First Design:
The FastAPI backend provides OpenAPI documentation, request validation, and RESTful endpoints. This enables easy integration with other systems.

PERFORMANCE CHARACTERISTICS

Response Latency:
End-to-end response time averages 2-3 seconds, breaking down as:
- Vector search: 50-100ms
- Embedding generation: 300-500ms
- LLM inference: 1-2 seconds
- API overhead: 100-200ms

Document Processing:
The system processes approximately 10 chunks per second. A 50-page PDF document takes about 15-20 seconds to fully process and index.

Scalability:
The stateless backend design enables horizontal scaling. Multiple API instances can run concurrently, sharing the same Pinecone index and Redis store. The free tier supports 10-20 concurrent users comfortably.

DEPLOYMENT STRATEGY

The system is designed for cloud deployment with minimal cost:

Development Environment:
Docker Compose orchestrates all services locally, including a Redis container. This provides a consistent development experience matching production.

Production Deployment:
The recommended deployment uses Render.com for the backend and Hugging Face Spaces for the frontend. This combination provides:
- Zero monthly cost
- Automatic HTTPS
- Auto-deployment from GitHub
- Built-in monitoring

Alternative deployments on Railway, AWS, or Azure are also supported through the provided Docker configurations.

SECURITY CONSIDERATIONS

Configuration Security:
All secrets are managed through environment variables. The .env file is excluded from version control. A .env.example template guides configuration without exposing credentials.

Input Validation:
File uploads are restricted by type and size. Pydantic models validate API requests. The system sanitizes all user inputs before processing.

Container Security:
Docker containers run as non-root users. The minimal Python slim base image reduces attack surface. Health checks ensure service availability.

OPERATIONAL ASPECTS

Logging:
Loguru provides structured logging with automatic rotation. Logs are retained for 10 days with a 500MB size limit per file. Log levels are configurable through environment variables.

Monitoring:
The /health endpoint reports status for all service dependencies. The frontend displays real-time connection status. Graceful degradation ensures partial functionality if services are unavailable.

Error Handling:
Custom exception classes categorize errors. API endpoints return appropriate HTTP status codes. User-facing messages are friendly while detailed errors go to logs.

TECHNICAL DECISIONS

Why Groq Over OpenAI:
- 10x faster inference (optimized hardware)
- Free tier with no credit card
- Comparable quality for RAG tasks
- Lower latency improves UX

Why Local Embeddings:
- Zero API costs
- No rate limits
- Data privacy (no external API calls)
- Consistent performance

Why Pinecone:
- Production-grade vector database
- Low latency (<100ms)
- Free tier sufficient for demos
- Easy integration with LangChain

Why Redis:
- Fast in-memory storage
- Automatic expiration (TTL)
- Industry standard
- Graceful fallback if unavailable
